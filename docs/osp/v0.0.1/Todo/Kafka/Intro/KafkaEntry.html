1. Event Streaming

  • 事件流以事件流的形式(in form of streams of event)从db、sensor、
    mobile device、cloud service和software app等event source
    实时捕获(capturing)数据；持久durably存储事件流以供后续检索retrieval；

  • 实时及回顾性(retrospectively)操纵manipulating、处理processing
    和响应reacting事件流；及按需将事件流路由routing到不同的目的地技术；

  • 事件流确保数据的连续流动continuous flow和解释interpretation，
    使正确的信息在正确的时间、正确的地点出现；

  • 事件记录发生的事情，在文档中也称为记录record或消息message，
    当向kafka读取或写入数据时，以事件形式完成，事件有键key，值value，
    和时间戳timestamp，另可选包含元数据头metadata header；

  • Producer是写入Kafka事件的客户端应用，Consumer是订阅(读取和处理)Kafka事件的用户，
    Producer和Consumer完全解耦，Kafka提供各种保证，如(处理事件)精确一次exactly-once；

  • Event按主题Topic组织和持久存储，主题类似于文件夹，事件是主题中的文件，
    主题通常是多生产者multi-producer和多消费者multi-subscriber；

  • 单主题可有零个，一个或多个向其写入事件的生产者，可以有零个，一个或多个订阅这些事件的消费者，
    可按需随时读取主题中的事件，与传统的消息系统不同，事件在消费consumption后不会被删除；

  • 但可通过主题的配置来定义Kafka应保留retain事件多长时间，之后旧事件将被丢弃discard，
    Kafka的性能在数据大小方面恒定effectively constant，故长时间存储数据非常好；

  • 主题是分区Partition的，即一个主题分布在位于不同Kafka broker的多个"bucket"上，
    允许客户端应用同时从多个代理读取数据和向多个代理写入数据；

  • 当一个新事件发布到一个主题时，会被追加append到该主题的一个分区，具有相同EventKey(如客户Id)
    的事件被写入同一个分区，Kafka保证给定主题分区的任何消费者将始终以与写入时完全相同的顺序读取该分区的事件；

  • 为具有容错性fault-tolerant和高可用性highly-available，
    每个主题都可复制，可跨地理区域或数据中心复制，常见的生产设置是复制系数为3，
    即始终有三份数据副本，此复制在topic-partition级别执行；

3. Use Case

  • Commit Log：类似Apache Bookeeper

  • Event Sourcing：事件溯源是应用设计风格，其状态的更改被记录为按时间顺序排列的记录序列；

  • Stream Processing：其它流处理框架如Storm，Samza；

  • Log Aggregation：其它日志聚合框架：Scribe或Flume

  • Metric：如运维监控数据，聚合来自分布式应用的统计信息，以生成操作数据的摘要；

  • Messaging：其它消息代理如：ActiveMQ，RabbitMQ，RocketMQ；

  • 实时处理支付和金融交易，如证券交易所、银行和保险；

  • 实时跟踪和监控汽车、卡车、车队和货运，如物流和汽车行业；

  • 持续捕获和分析来自物联网设备(如工厂和风电场)的传感器数据；

  • 收集客户互动和订单并立即做出反应react，如零售、酒店和旅游业及移动应用；

  • 监测医院护理中的患者并预测病情变化，确保在紧急情况及时治疗；

  • 连接、存储公司不同部门产生的数据并使其可用；

  • 作为数据平台、事件驱动架构和微服务的基础；
