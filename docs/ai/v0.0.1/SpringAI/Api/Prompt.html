1. Introduction

  • Prompt是指导AI模型生成特定输出的输入，这些提示的design and phrasing
    会显著影响significantly influence模型的响应response；

  • Spring AI中handling prompt类似于Spring Mvc的View，
    这涉及到为动态内容创建带有占位符的大量文本，再根据用户请求或应用中的其它代码
    替换占位符，另一个类比是SQL语句，它包含某些表达式的占位符；

  • Spring AI引入更高级别抽象与AI模型交互，此章节的基础类类似于Jdbc，如ChatClient类
    类似于Jdk中Jdbc库，在此基础上，Spring AI可提供类似于JdbcTemplate的辅助类、
    Spring数据存储库，并最终提供更高级的构造，如考虑过去与模型交互的ChatEngines和Agent；

  • Prompt的结构在AI领域随着时间的推移而演变，最初Prompt仅是简单的字符串，
    逐渐包括特定输入的占位符，如AI模型识别的"USER:"；

2. Api Intro

2.1. Prompt

  • 用ChatClient.generate()获取Prompt实例并返回ChatResponse

  • Prompt类function as一系列有组织的Message对象的容器，每个对象构成整个提示的一段
    segment，每条消息在提示中都体现一个独特的角色，其内容content和意图各不相同intent，

  • 这些角色可以包含各种元素，从用户查询到AI生成的响应或相关背景信息，这能与AI模型进行
    复杂而详细的交互，因为提示是由多条消息构建的，每条消息都被指定在对话中扮演特定的角色

public class Prompt {

    private final List<Message> messages;

    //constructor and utility method
}

2.2. Message

  • Message接口封装一条文本消息、一组属性(如Map)和
    一个称为MessageType的分类categorization；

  • Message接口的各种实现对应于AI模型可处理的不同类别的消息；

public interface Message {

   String getContent();

   Map<String, Object> getProperties();

   MessageType getMessageType();

}

3. Role

  • AI Prompt演变从基本、直接的文本过渡到具有特定角色和结构的更有组织和复杂的格式，
    最初是简单的字符串，文本行，逐渐变成字符串中包括特定占位符如"USER:"，
    AI模型可识别并相应做出响应，朝更加结构化的提示迈出一步；

  • OpenAI随后引入更有组织性的方法，提示不仅仅是单个字符串，而是一些列消息，每条消息仍然是文本形式，
    都分配特定角色，这些角色对消息分类，澄清AI模型提示的每个片段的上下文和目的purpose，
    此结构化方法增强与AI能沟通的细微差别与有效性，因提示的每个部分在互动中都扮演独特而明确的角色；

3.1. System Role

  • 系统角色指导AI的行为和响应风格，设置AI如何解释和回复输入的参数或规则，如对话前向AI提供指令；

3.2. User Role

  • 用户角色代表用户输入：他们对AI的问题，命令或陈述，此角色是基本的，因构成AI响应的基础；

3.3. Assistant Role

  • 助理角色：AI对用户输入的响应，这不仅是答案answer或反应reaction，保持对话的流畅性，
    通过跟踪AI前的响应(其助理角色信息)，该系统确保连贯和上下文相关的互动；

3.4. Function Role

  • 职能角色处理会话中特定任务或操作，当系统角色设定AI的整体行为时，职能角色专注于
    执行用户要求的某些动作或命令，如AI中的某特殊功能，在需执行特定功能时使用，如计算，
    获取数据或其它任务，而不仅仅是说话，职能角色使AI能在对话回应之外提供实际帮助；

public enum MessageType {

	USER("user"),

	ASSISTANT("assistant"),

	SYSTEM("system"),

	FUNCTION("function");

	private final String value;

	MessageType(String value) {
		this.value = value;
	}

	public String getValue() {
		return value;
	}

	public static MessageType fromValue(String value) {
		for (MessageType messageType : MessageType.values()) {
			if (messageType.getValue().equals(value)) {
				return messageType;
			}
		}
		throw new IllegalArgumentException("Invalid MessageType value: " + value);
	}

}

3.5. PromptTemplate

  • PromptTemplate是关键组件类，此类使用StringTemplate引擎来构造constructing
    和管理managing提示，PromptTemplate类旨在促进创建结构化提示，并将提示发送到AI模型以处理；

public class PromptTemplate implements PromptTemplateActions,
	PromptTemplateMessageActions {

}

  • PromptTemplateStringActions：专注于创建creating
    和呈现rendering提示字符串，表示最基本的提示生成形式；

  • PromptTemplateMessageActions：
    为通过生成和操作Message对象来创建提示而定制；

  • PromptTemplateActions：
    返回Prompt对象，该对象可传递给ChatClient以生成响应；

public interface PromptTemplateStringActions {

	String render();

	String render(Map<String, Object> model);

}

  • render()：将提示模板渲染为最终的字符串格式，无需外部输入，适用于没有占位符或动态内容的模板；

  • render(Map<String, Object> model)：增强渲染功能以包括动态内容，
    Map的key为提示模板的占位符名词，值是要插入的动态内容；

public interface PromptTemplateMessageActions {

	Message createMessage();

	Message createMessage(Map<String, Object> model);

}

  • createMessage()：创建不带额外additional数据的Message对象，
    用于静态或预定义的消息内容；

  • createMessage(Map<String, Object> model)：扩展createMessage()，
    以集成动态内容，接受Map，每个entry表示消息模板中占位符及响应的动态值；

public interface PromptTemplateActions extends PromptTemplateStringActions {

	Prompt create();

	Prompt create(Map<String, Object> model);

}

  • create()：生成没有外部数据输入的Prompt对象，适合静态或预定义的提示；

  • create(Map<String, Object> model)：扩展create()，
    以包括动态内容，Map的每个entry为提示模板中的占位符和关联的动态值；

5. Prompt Engineering

  • 在AGI中，提示的创建对开发至关重要，提示的质量和结构会显著影响AI输出的有效性，
    在设计深思熟虑的提示方面投入时间和经历可极大提高AI效果；

  • Sharing and Discussing提示是AI社区的常见做法，
    这种协作方法创造共享学习环境，且可识别和使用高效的提示；

  • 这一领域的研究通常包括分析和比较不同的提示，以评估其在各种情况下的有效性；

  • 掌握Grasping最有效的提示使用，AI技术快速发展是持续挑战，
    应认识到提示工程的重要性，并考虑利用社区和研究的见解来改进即时创作策略

5.1. Effective Prompt

  • Instruction：向A提供清晰直接的说明，类似于你与人沟通的方式，
    这种清晰度对帮助AI"understand"预期内容至关重要；

  • External Context：必要时包括相关背景信息或AI响应的具体指导，
    这种外部环境构成提示的框架，并有助于AI掌握整体场景；

  • User Input：最直接，用户的直接请求或问题构成提示的核心；

  • Output Indicator：很棘手，涉及到AI响应指定的格式，
    如json，但AI并不能总严格遵守此格式，如"这是你的json"；

6. Simple Technique

  • https://www.promptingguide.ai/introduction/examples.en

  • Text Summarization：将大量extensive文本简化为简洁的concise摘要，
    捕捉关键点和主要思想，同时省略不太关键的细节；

  • Question Answering：重点是根据用户提出的问题，从提供的文本中得出具体的答案，
    它是关于精确定位pinpointing和提取相关extracting relevant信息以响应查询；

  • Text Classification：系统地Systematically将文本分类为
    预定义的类别category或组，分析文本并根据其内容将其分配到最合适的类别；

  • Conversation：创建交互式对话，AI可与用户进行来回
    (back-and-forth)交流，模拟自然的对话流程；

  • Code Generation：根据特定的用户需求或描述生成
    功能代码片段，将自然语言指令转换为可执行代码

8. Token

  • Token在AI模型处理文本方式中至关重要，充当桥梁，将单词word转换为AI模型可处理的格式，
    转换分两阶段：A：输入时将单词转换为标记word transform into token upon input，
    B：再在输出中将这些标记转换回单词token convert back into word in output；

  • 标记化Tokenization：即将文本分解为标记的过程breaking down text
    into token，是AI模型理解和处理语言的基础，AI模型用这种标记化格式来理解和响应提示；

  • 可将标记理解为单词的一部分portion of word，通常通常一个标记代表一个单词的3/4，
    如莎士比亚的全集，总计约90万字，将翻译成约120万个token；

  • 标记再计费billing和模型能力model capability方面也很有实际意义；

    • Billing：AI模型服务通常根据token使用情况计费，输入(提示)
      和输出(响应)都有助于总token计数，使较短的提示更具成本效益；

    • Model Limit：不同AI模型有不同token限制，定义其"Context Window"，
      它们一次可处理的最大信息量，如GPT-3限制是4K token，而其他型号如Claude 2
      和Meta Llama 2限制是100K token，一些研究型号最多可处理100万个token；

    • Context Window：：模型的标记限制决定其上下文窗口，超过此限制的输入
      不由模型处理，只发送最少量的有效信息进行处理很重要，如当询问《哈姆雷特》时，
      没必要包括莎士比亚所有其他作品中的token；

    • Response Metadata：AI模型的响应元数据包括使用的token数量，
      这是管理使用和成本的重要信息；
