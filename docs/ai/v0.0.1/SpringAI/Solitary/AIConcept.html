1. 概览

  • Spring AI灵感来自于Python，如LangChain和LlamaIndex，
    不仅仅面向Python开发人员；支持的向量数据库如：Chroma，
    Milvus，Neo4j，Postgres Vector，Redis，Weaviate；

  • Spring AI的核心提供抽象，抽象有多个实现，如ChatClient接口；
    还提供更高级功能，以解决常见用例，如QA over Doc，Chat with Doc；

2. Model

  • AI模型：用来处理和生成信息的算法，通常模仿人类的认知功能，
    mimicking human cognitive function；

  • 通过从大型数据集中学习模式和见解(pattern and insight)，
    模型可进行预测，文本，图像和其它输出，增强跨行业的各种应用程序；
    enhancing various application across industry；

3. Prompt

  • Prompt用作指导AI模型产生特定输出的基于语言的输入的基础；
    serve as foundation for language-based input
    that guide AI model to produce specific output；

  • 如ChatGPT中，Prompt可能只是发送到Api的对话框中输入的文本，
    在许多AI模型中，提示的文本不仅仅是简单的字符串；

  • ChatGPT的Api在中有多个文本输入，每个文本输入都被分配一个角色Role，
    如有系统角色，它告诉模型如何行为(behave)，并为交(interaction)
    互设置上下文，还有用户角色，它通常(typically)是来自用户的输入；

  • 精心制作有效的提示是艺术和科学，ChatGPT是与人类对话而设计，
    与SQL提问截然不同，人必须与AI模型沟通，如与小明交谈；
    crafting effective prompt both art and science；

4. Prompt Template

  • 创建有效的提示涉及建立请求的上下文并用特定于用户输入的值替换部分请求；

  • effective prompt involve establishing context of request and
    substituting part of request with value specific to user input；

  • 此过程用传统基于文本的模板引擎进行提示创建和管理，
    traditional text-based template engine；

  • {adjective} joke about {content}；

  • 提示模板类似于SpringMvc中的View，提供模型对象(通常是Map)来填充模板内的占位符；

  • 发送到模型的提示的具体数据格式有相当大的变化，提示可包含多个消息，
    每条消息中每个字符串代表模型中不同的角色；

  • considerable variability in specific data format
    of prompt sent to model，where each str in
    each msg represent distinct role for model；

5. Embedding

  • Embeddings将文本转换为数值数组或向量，使AI模型能处理和解释语言数据，
    这种从文本到数字的转换是AI如何与人类语言交互并理解人类语言的关键要素；

  • Embeddings transform text into numerical array or vector，
    enabling AI model to process and interpret language data，

  • transformation from text to number and back as key element
    in how AI interact with and understand human language；

  • 作为探索(exploring)AI的Java开发人员，
    没必要理解复杂的数学理论或这些向量表示背后的具体实现；

  • not necessary to comprehend intricate
    mathematical theory or specific
    implementation behind these vector representation；

  • 基本了解它们在AI系统中的角色和功能就足够，特别是当我们将AI功能集成到应用程序中时；

  • basic understanding of role and function within
    AI system suffice，particularly when integrating
    AI functionality into application；

  • Embeddings在实际应用中特别相关，如检索增强生成模式，
    它们能将数据表示为语义空间中的点，
    该语义空间类似于欧几里德几何的二维空间，但维度更高；

  • Embeddings are particularly relevant in practical
    application like retrieval-augmented generation pattern，
    enable representation of data as point in semantic space，
    which is akin to 2D space of Euclidean geometry，
    but in higher dimension，

  • 这意味就像欧几里得几何中平面上的点根据其坐标可以接近或远离一样，
    在语义空间中，点的接近反映了含义的相似性
    故关于相似主题的句子在这个多维空间中的位置更接近，

  • just like how point on plane in Euclidean geometry
    can be close or far based on their coordinate；
    in semantic space，proximity of point reflect
    similarity in meaning，so sentence about similar
    topic are positioned closer in this multi-dimensional
    space，much like point lying close to each other on graph；

  • 就像图表上彼此靠近的点一样。 这种接近性有助于文本分类、语义搜索甚至产品推荐等任务，
    因它允许AI根据相关概念在扩展语义环境中的"位置"来辨别和分组；

  • proximity aid in task like text classification，
    semantic search and even product recommendation，
    as it allow AI to discern and group relate concept
    based on 'location' in this expanded semantic landscape.

  • 可把语义空间想象成向量
    we can think of semantic space as vector

6. Token

  • token serve as building block of how AI model work，
    on input，model convert word to token，
    on output,they convert token back to word；

  • token是AI模型工作的构建块，输入时，模型将word转换为token，
    输出时，将token换回word；

  • 在英语中，一个token对应一个word的75%，莎士比亚全集约90w字，
    等价于120w个token；token = $

  • 托管AI模型的背景下，费用由使用的token决定，输入和输出会影响总token数；
    另模型还受token限制，这限制单个Api调用中处理的文本量，
    此阈值通常称为上下文窗口，该模型不会处理任何超过此限制的文本；

  • model subject to token limit，
    threshold referr to as 'context window'；

  • ChatGPT3有4K的token限制，GPT4提供不同的选项，如8K，16K和32K，
    Claude AI模型有100K的token限制，Meta有1M的token限制AI模型；

  • 要使用GPT4总结莎士比亚作品集，需设计(devise)软件工程策略来分割(chop up)数据，
    并在模型的上下文窗口限制内呈现数据，可使用Spring AI完成此任务；

7. Output Parsing

  • AI模型的传统输出是String，即便要求回复采用json形式；
    输出解析采用精心设计的提示，通常需与模型多次交互才能实现所需的格式；
    OpenAI引入OpenAI函数(function)作为精确指定模型所需输出格式的方法；

  • output parsing employ meticulously crafted prompt，
    often necessitating multiple interaction with model
    to achieve desired formatting，OpenAI Function；

8. Bringing Data to Model

  • 如何为AI模型配备(equip)未经训练的信息?
    有两种技术可自定义AI模型以合并(incorporate)数据；

  • 注：GPT3.5/4.0数据集仅延续到2021年9月，
    故该模型不知道该日期后知识的问题的答案，此数据集大约650G；

8.1. Fine Tuning

  • 微调：传统ML涉及裁剪(tailoring)模型并更改其内部权重，然而对于
    ML专家来说，这是具有挑战性的过程，且对GPT等模型来说，因其规模，资源极其密集
    (extremely resource-intensive)，另某些型号可能不提供此选项；

8.2. Prompt Stuffing

  • 提示填充：更实用的替代方案是将数据嵌入到提供给模型的提示中，
    考虑到模型的token限制，用填充提示在模型的上下文窗口中呈现相关数据；

8.3. Function Calling

  • 函数调用：大型语言模型LLM在训练后被冻结，导致知识陈旧stale，无法访问或
    修改外部数据；可使用函数调用机制解决，它允许注册自定义的、用户的函数，函数将LLM
    连接到外部系统的API，外部系统可向LLM提供实时数据，并代表LLM执行数据处理动作；

  • Spring AI库可帮助实现基于"填充提示"技术，
    也称检索增强生成(RAG：Retrieval Augmented Generation)解决方案；

9. RAG

  • RAG：旨在解决相关数据纳入提示中以实现准确的AI模型响应挑战；
    emerge to address challenge of incorporating relevant
    data into prompt for accurate AI model response；

  • 此方法涉及批处理风格的编程模型，作业(job)从文档(document)中读取非结构化数据，
    对其转换，然后写入向量数据库，从较高层次来看，这是etl流水线
    (Transform and Load) pipeline)，RAG技术的检索部分使用向量数据库；

  • 非结构化数据加载到向量数据库，将原始文档分割成更小的部分，分割有两步骤：

    • 将文档拆分为多个部分，同时保留内容的语义边界，如对包含段落和表格的文档，
      应避免在段落或表格的中间拆分文档，对代码，避免在方法实现的中间拆分代码；

    • split up document into part while
      preserving semantic boundary of content

    • 将文档的各部分进一步拆分为大小仅为AI模型令牌限制的小百分比；

    • split up document part further into part whose
      size as small percentage of AI Model token limit；

  • RAG的下一阶段是处理用户输入，当AI模型要回答用户的问题时，
    问题和所有"相似(similar)"文档片段都会被放入发送给AI的提示中，
    即使用向量数据库的原因，向量数据库非常擅长查找"相似"内容；

  • 实现RAG时使用概念，映射到Spring AI中的类：

    • DocumentReader：Java functional接口，负责从数据源加载
      List<Document>，常见的数据源如Pdf，Json，Markdown；

    • Document：数据源的基于文本的表示形式，还包含用于描述内容的元数据；
      text-based representation of data source
      that also contain metadata to describe content；

    • DocumentTransformer：负责以各种方式处理数据；
      (如将文档分割成更小的部分或向文档添加额外的元数据)

    • DocumentWriter：允许我们将文档保存(persist)
      到数据库中(最常见的时AI Stack，向量数据库)；

    • Embedding：将数据表示为List<Double>，
      向量数据库使用它来计算用户查询与相关文档的"相似度(similarity)"；

10. Evaluating AI Response

  • 有效评估(effectively evaluating)AI系统响应的用户请求的输出
    对确保最终应用的准确性(accuracy)和有效性(usefulness)很重要；

  • 此评估过程涉及分析(analyzing)生成的响应是否符合(align with)
    用户的意图(intent)和查询上下文(context of query)；

  • 相关性，连贯性和事实正确性等指标用于衡量AI能生成的响应的质量；
    relevance，coherence and factual correctness，
    used to gauge quality of AI generate response；

  • 一种方法论涉及呈现用户请求和AI模型对模型的响应，查询响应是否与提供的数据一致；
    另利用向量数据库中存储的信息作为补充数据可增强评估过程，有助于确定响应相关性；

  • presenting both user request and AI model response to
    model，querying whether response align with provide data；
