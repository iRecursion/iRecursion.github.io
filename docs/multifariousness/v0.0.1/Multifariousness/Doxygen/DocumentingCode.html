1. Special Comment Block

1.1. CLike Lang

类 C 语言 (C/C++/C#/Objective-C/PHP/Java) 的注释块
* 类C语言(C，Cpp，Java等)的注释块：

  • 对代码中的每个实体，有两种类型描述，他们共同构成该实体的文档：
    简要描述brief description和详细描述detail description；

  • 允许多个简要描述或详细描述，但不建议，因未指定描述的显示顺序；

  • 简要描述是简短文字，short one-liner，详细描述提供更长，更详细的文档；

  • 正文描述("in body" description)：可充当详细描述，也可描述一组实现细节；

  • 对Html输出，简要描述还用于在引用item的地方提供tooltip；

  • comment block as detail description：

  • A：使用Javadoc样式，由两个*开头的C样式注释块组成：

/**
 * JavaDoc Style
 */

  • B：使用QT样式，并在C样式注释块开头后添加感叹号(!)，中间的*是可选：

/*!
 * QT Style with intermediate asterisk
 */

/*!

 QT Style without intermediate asterisk

 */

  • C：选择使用至少两个Cpp注释行的块，其中每行都以单个额外斜杠
    或感叹号开头    ，此场景下：空白行会劫数文档块，两种示例如下：

///
/// at least two cpp comment line with additional slash
///


//!
//! at least two cpp comment line with exclamation mark
//!

  • D：注释块在文档中更显眼：

/*******************************************//**
 *  comment
 ***********************************************/

/////////////////////////////////////////////////
/// comment
/////////////////////////////////////////////////

/***********************************************
 * comment
 ***********************************************/

  • 注：2个斜线结束正常注释块并开始特殊注释块；

  • 注：用clang-format等重新格式化程序时要小心，因它
    可能会将此类注释视为2个单独的注释并在它们之间引入空格；

  • E：banner comment：JAVADOC_BANNER 设置为YES：
    https://www.doxygen.nl/manual/config.html#cfg_javadoc_banner
    https://www.doxygen.nl/manual/examples/javadoc-banner/html/javadoc-banner_8h.html

/**
 * brief history of JavaDoc-style (C-style) comment
 *
 * typical JavaDoc-style C-style comment，start with two asterisk
 *
 * @param theory Even if there only one possible unified theory，
 * it is just a set of rules and equations.
 */
void cstyle( int theory );

/******************************************************************************
 * brief history of JavaDoc-style (C-style) banner comment
 *
 * typical JavaDoc-style C-style "banner" comment，
 * start with forward slash follow by some number,
 *
 * n of asterisk，where n > 2，It's written this way to
 * be more "visible" to developer who are reading source code
 *
 * often，developer are unaware that this is not (by default)
 * valid Doxygen comment block!
 *
 * however，as long as JAVADOC_BLOCK = YES add to Doxyfile，
 * it will work as expected；
 *
 * this style of commenting behave well with clang-format，
 *
 * @param theory even if there only one possible unified theory，
 * it is just a set of rules and equations.
 ******************************************************************************/
void javadocBanner( int theory );

/**************************************************************************//**
 * brief history of Doxygen-style banner comment
 *
 * Doxygen-style C-style "banner" comment，
 * It start with "normal" comment and then convert to
 *
 * "special" comment block near the end of the first line，
 * It is written this way to be more "visible" to developer
 * who are reading the source code；
 *
 * this style of commenting behave poorly with clang-format；
 *
 * @param theory even if there is only one possible unified theory，
 * it is just a set of rules and equations.
 ******************************************************************************/
void doxygenBanner( int theory );

  • brief description several possibility：

  • A：可将[fuchsia]#\brief#命令与上述注释块之一结合使用，
    此命令在段落末尾结束，故详细描述位于空行之后；

/*! \brief Brief description,
 *         Brief description continue；
 *
 *  detail description start from here
 */

  • B：若在配置文件中将JAVADOC_AUTOBRIEF设置为YES，则用Javadoc
    样式注释块将自动开始简短描述，该描述以第一个点结尾，后跟空格或换行符：

  • https://www.doxygen.nl/manual/config.html#cfg_javadoc_autobrief

/** Brief description which end at this dot. detail follow
 *  here.
 */

  • 该选项对多行特殊 C++ 注释具有相同的效果：

/// Brief description which end at this dot. Details follow
/// here.

  • C：用特殊的C++样式注释，该注释不超过一行：

/// 简要说明。
/** 详细说明。 */
//! 简要说明

//! 详细说明
//! 从这里开始

  • 注：上面示例中的空白行，该空白行用于将简要说明与包含详细说明的块分开，
    此时需将JAVADOC_AUTOBRIEF设置为NO；

  • D：多个详细描述：

//! Brief description, which is
//! really a detailed description since it spans multiple lines.
/*! Another detailed description!
 */

  • 与大多文档系统不同：Doxygen允许将成员(包括全局函数)的文档放在定义之前；

  • 这样文档就可放在源文件中，而不是头文件中，这可使头文件保持紧凑，
    并允许成员的实现者更直接地访问文档，作为权衡，
    可将简短描述放在声明之前，将详细描述放在成员定义之前

1.1.1. Put Doc After Member

  • 若想要文档化file，struct，union，class或enum的member，
    有时需将文档块放在成员之后而不是之前；

  • 为此，必须在注释块中放置一个额外的 < 标记，注：这也适于函数的参数；

 
int var; /*!< detail description after member */
int var; /**< detail description after member */
int var; //!< detail description after member
         //!<
int var; ///< detail description after member
         ///<

  • brief description after member

int var; //!< brief description after member
int var; ///< brief description after member

  • 对函数，可用@param命令来文档化参数，
    然后用[in]、[out]、[in,out]来document direction；

  • 对内联inline文档，可通过从direction属性开始来实现，如：

void foo(int v /**< [in] doc for input parameter v */);

  • 注：这些块具有与上一节中特殊注释块相同的结构和含义，只有 < 表示成员位于块前面而不是块后面；

/*! A test class */

class AfterdocTest{
  public:
    /** An enum type.
     *  The documentation block cannot be put after the enum!
     */
    enum EnumType
    {
      int EVal1,     /**< enum value 1 */
      int EVal2      /**< enum value 2 */
    };
    void member();   //!< a member function.

  protected:
    int value;       /*!< an integer value */
};

  • https://www.doxygen.nl/manual/examples/afterdoc/html/class_afterdoc___test.html

  • 这些块只能用于文档化member and parameter，它们不能用于文档化file，
    class，union，struct，group，namespace，macro，enum本身；

  • 另下一节中提到的结构命令，如class不允许在这些注释块内使用，
    将此构造用作宏定义的一部分时要小心；

  • 因当在应用宏的地方将MACRO_EXPANSION设置为YES时，注释也将被替换，
    然后此注释将用作遇到的最后一项的文档，而不是宏定义本身的文档!

1.1.2. Example

//!  A test class.
/*!
  A more elaborate class description.
*/

class QTstyle_Test
{
  public:

    //! An enum.
    /*! More detailed enum description. */
    enum TEnum {
                 TVal1, /*!< Enum value TVal1. */
                 TVal2, /*!< Enum value TVal2. */
                 TVal3  /*!< Enum value TVal3. */
               }
         //! Enum pointer.
         /*! Details. */
         *enumPtr,
         //! Enum variable.
         /*! Details. */
         enumVar;

    //! A constructor.
    /*!
      A more elaborate description of the constructor.
    */
    QTstyle_Test();

    //! A destructor.
    /*!
      A more elaborate description of the destructor.
    */
   ~QTstyle_Test();

    //! A normal member taking two arguments and returning an integer value.
    /*!
      \param a an integer argument.
      \param s a constant character pointer.
      \return The test results
      \sa QTstyle_Test(), ~QTstyle_Test(), testMeToo() and publicVar()
    */
    int testMe(int a,const char *s);

    //! A pure virtual member.
    /*!
      \sa testMe()
      \param c1 the first argument.
      \param c2 the second argument.
    */
    virtual void testMeToo(char c1,char c2) = 0;

    //! A public variable.
    /*!
      Details.
    */
    int publicVar;

    //! A function variable.
    /*!
      Details.
    */
    int (*handler)(int a,int b);
};

  • https://www.doxygen.nl/manual/examples/qtstyle/html/class_q_tstyle___test.html

  • https://www.doxygen.nl/manual/config.html#cfg_brief_member_desc

  • https://www.doxygen.nl/manual/config.html#cfg_repeat_brief

  • https://www.doxygen.nl/manual/config.html#cfg_javadoc_autobrief

  • brief description包含在类、命名空间或文件的成员概述中，
    并使用小斜体字体打印(可将BRIEF_MEMBER_DESC设置为NO来隐藏此说明)；

  • 默认：简要说明成为详细描述的第一句，但可通过将REPEAT_BRIEF
    设置为NO来更改，对Qt样式，简要说明和详细描述都是可选；

  • 默认：Javadoc样式文档块的行为与Qt样式文档块相同，但这不符合Javadoc规范，
    其中文档块的第一句会自动被视为简要说明，用JAVADOC_AUTOBRIEF=YES来启用；

  • 若启用此选项并想在句子中间放置一个点而不结束它，
    则应该在它后面放置一个反斜杠和一个空格，若：

  /** Brief description (e.g.\ using only a few words). Details follow. */

  • 类似地，若希望Qt风格文档块的第一句话自动被视为简短描述，
    将QT_AUTOBRIEF设置为YES；

/**
 *  A test class. A more elaborate class description.
 */

class Javadoc_Test
{
  public:

    /**
     * An enum.
     * More detailed enum description.
     */

    enum TEnum {
          TVal1, /**< enum value TVal1. */
          TVal2, /**< enum value TVal2. */
          TVal3  /**< enum value TVal3. */
         }
       *enumPtr, /**< enum pointer. Details. */
       enumVar;  /**< enum variable. Details. */

      /**
       * A constructor.
       * A more elaborate description of the constructor.
       */
      Javadoc_Test();

      /**
       * A destructor.
       * A more elaborate description of the destructor.
       */
     ~Javadoc_Test();

      /**
       * a normal member taking two arguments and returning an integer value.
       * @param a an integer argument.
       * @param s a constant character pointer.
       * @see Javadoc_Test()
       * @see ~Javadoc_Test()
       * @see testMeToo()
       * @see publicVar()
       * @return The test results
       */
       int testMe(int a,const char *s);

      /**
       * A pure virtual member.
       * @see testMe()
       * @param c1 the first argument.
       * @param c2 the second argument.
       */
       virtual void testMeToo(char c1,char c2) = 0;

      /**
       * a public variable.
       * Details.
       */
       int publicVar;

      /**
       * a function variable.
       * Details.
       */
       int (*handler)(int a,int b);
};

1.1.3. Doc At Other Place

  • 在上一节的示例中，注释块始终位于file，class，namespace
    的声明或定义之前，或位于其成员之一之前或之后；

  • 虽这通常很方便，但有时可能需要将文档放在其他地方，
    对文档化文件，这甚至是必需的，因没有"在文件前面"这样的说法；

  • Doxygen允许将文档块放在几乎任何地方，函数主体内部或普通C样式注释块内部除外；

  • 不将文档块直接放在项目之前或之后的代价是需在文档块内放置结构命令，
    这会导致一些信息重复，故除非其他要求迫使这样做，否则应避免使用结构命令；

  • 结构命令(同其它命令)以反斜杠(\)开头，或若喜欢Javadoc样式，
    则以at符号(@)开头，后跟命令名称和一或多个参数；

  • https://www.doxygen.nl/manual/commands.html#cmd_intro

  • 如若想文档化上面示例中的Test类，那可将以下文档块放在Doxygen读取的输入中的某个位置：

 
/*! \class Test
    \brief A test class.

    A more detailed class description.
*/
